Activation of [[mediacapture-extensions]]'s [[[mediacapture-extensions#media-capture-html-elements|<usermedia>]]] element should only be allowed by an intentional click (or other gesture) by the user. Programmatic activation or tricking a user in activating the element ("clickjacking") should not activate the element. This note proposes specific mechanisms to accomplish this, based on the original [[pepc]] design. The intent of this note is to provide a basis of discussion for standardization of these measures.

This is a purely informational note.

Introduction

[[mediacapture-extensions]]' definition of the trusted algorithm contains two notes, reproduced here:

  • At the top of the trusted algorithm:

    HTMLUserMediaElement stream creation is supposed to happen by genuine user action. User agents should suppress activation for "click-jacking", as well as for programmatic activation. Since exact critera for these are difficult to specify, we presently leave this as implementation defined. User agents must at minimum reject untrusted events.

  • In trusted, step 2, following a recommendation to run additional, implementation-defined steps:

    Additional implementation guidelines will be provided.

This document proposes those guidelines.

Integration with <usermedia> element.

The usermedia element's initialization steps should also:
  1. Initialize the internal {{ActivationBlocker/[[BlockerList]]}} to «».
  2. Initialize the internal {{ActivationBlocker/[[IntersectionRect]]}} to undefined.
  3. Initialize the internal {{ActivationBlocker/[[IntersectionObserver]]}} to undefined.
The usermedia element's [=insertion steps=] should also:
  1. Initialize the internal {{ActivationBlocker/[[BlockerList]]}} to «».
  2. [=set/Append=] [=this=] to [=node navigable=]'s {{navigable/[[ActivationBlockableElements]]}}.
  3. Initialize the internal {{ActivationBlocker/[[IntersectionRect]]}} with undefined.
  4. Initialize the internal {{ActivationBlocker/[[IntersectionObserver]]}} with the result of constructing a new {{IntersectionObserver}} with [=ActivationBlocker/IntersectionObserver callback=] and «[ "{{IntersectionObserverInit/rootMargin}}" → `"-4px"` ]».
  5. Call {{ActivationBlocker/[[IntersectionObserver]]}}.observe([=this=]).
  6. [=ActivationBlocker/Add an expiring blocker=] with reason {{ActivationBlockerBlockerReason/recently_attached}}.
  7. If the [=navigable/traversable navigable=] of the [=node navigable=] of [=this=] is a [=fenced navigable container/fenced navigable=], then [=ActivationBlocker/add a permanent blocker=] with {{ActivationBlockerBlockerReason/illegal_subframe}}.
The usermedia element's [=removing steps=] should also:
  1. [=list/Remove=] [=this=] from [=node navigable=]'s {{navigable/[[ActivationBlockableElements]]}}.
The trusted algorithm, step 2, currently reads: User agents should run additional, implementation-defined steps to determine trusted-ness of |event|. Replace this step with the following step:
  1. If [=this=]'s [=ActivationBlocker/blocker=] is not nothing, return.
usermedia does not provide a hook to observe changes to its styling properties. While the usermedia element is inserted into a document, i.e. between its [=insertion steps=] and [=removing steps=], it should [=ActivationBlocker/check the style=] of the element whenever its styling properties change, including the inherited ones.

Activation Blockers

To guarantee meaningful user intent, we will assume that an activation is — generally — what the user means to do. We also define circumstances that where the user agent assumes non-intentional activation, and call these activation blockers. A usermedia element then maintains a set of current activation blockers, and will disregard an activation if the set of current blockers is non-empty.

A common "click-jacking" technique is to provide the user with something interesting to click on, and then, when a click is likely to occur, to programmatically move the page so that a different element is now in the position that the user is likely to click on. To prevent this, the user agent will add an activation blocker that expires after 500 ms whenever the usermedia element is moved.

Internal Slots

Supporting activation blockers requires the following internal slots at the element:

[[BlockerList]]
A list of records, containing a blocker timestamp and a blocker reason. The [=ActivationBlocker/blocker reason=] is a {{ActivationBlockerBlockerReason}}.
[[IntersectionObserver]]
A reference to an IntersectionObserver.
[[IntersectionRect]]
A {{DOMRectReadOnly}} that stores the most recently seen intersection, i.e. the position of the element relative to the viewport.

To prevent a page from "spamming" the user with a large number of elements, we'd like to keep track of the number of active elements. To accomplish this, the [=/navigable=] maintains an [=ordered set=] of elements, [[ActivationBlockableElements]]. This [=ordered set=] is used to evaluate the [=blockers=] of type {{ActivationBlockerBlockerReason/unsuccessful_registration}}.

Activation Blockers, Blocker Reasons, and Blocker Lifetimes

The key goal of the elements in this specification is to reflect a user's conscious choice, and we need to make sure the user cannot easily be tricked into activating it. To do so, we maintain a list of blocker reasons, which may - permanently or temporarily - prevent the element from being activated. 

          enum ActivationBlockerBlockerReason {
            "illegal_subframe", "unsuccessful_registration",
            "recently_attached", "intersection_changed",
            "intersection_out_of_viewport_or_clipped",
            "intersection_occluded_or_distorted", "style_invalid"
          };

These blockers come with three lifetimes: Permanent, temporary, and expiring.

Permanent blocker
Once an element has a permanent blocker, it will be disabled permanently. There are used for issues that the website owner is expected to fix. An example is an element inside a [^fencedframe^].
Temporary blocker
This is a blocker that will only be valid until the blocking condition no no longer occurs. An example is an element that is not currently in view. All [=temporary blockers=] turn into [=expiring blockers=] once the condition no longer applies.
Expiring blocker
This is a blocker that is only valid for a fixed period of time. This is used to block abuse scenarios like "click jacking". An example is an element that has recently been moved.

Styling, and Styling Restrictions

usermedia elements constrain the styling that can be applied to them. These are expressed as conditions on their styling, and come in two flavours:

  • If the condition isn't met, the element is blocked.
  • A [=user-agent style sheet=], which enforces certain styling.
    Some style conditions cannot easily defined in a standard CSS stylesheet, for example because we want to express styling conditions based on the computed style, rather than the declared style.
To check the style of a usermedia element:
  1. Check the conditions in [[[#style-blockers]]]. Let |block| be a boolean, which is true when those conditions are met, and false otherwise.
  2. If |block|, then [=ActivationBlocker/add a temporary blocker=] with reason {{ActivationBlockerBlockerReason/"style_invalid"}}.
  3. Otherwise, [=ActivationBlocker/remove blockers=] with reason {{ActivationBlockerBlockerReason/"style_invalid"}}.

Blocking conditions

If the following conditions aren't met, a {{ActivationBlockerBlockerReason/"style_invalid"}} [=temporary blocker=] is added:

color
background-color
Set by default to the user agent's default `button` colors. The contrast ratio between the 2 colors needs to be at least 3. Alpha has to be 1.
font-size
If [=specified value=] is expressed as absolute-size:
  • Then font-size must be between small and xx-large.
  • Otherwise, the [=computed value=] must be larger than or equal to the [=computed value=] for small and less than or equal to the [=computed value=] of xx-large.
Define "alpha" and "contrast ratio".

User-agent defined stylesheet

A usermedia element is expected to render with the following [=user-agent style sheet=]: 

          @namespace "http://www.w3.org/1999/xhtml";
          usermedia {
            opacity: 1.0;
            line-height: normal !important;
            white-space: nowrap !important;
            user-select: none !important;
            appearance: auto !important;
            box-sizing: content-box !important;
            vertical-align: middle !important;
            text-emphasis: initial !important;
            text-shadow: initial !important;
          }

Additional, stylesheet-like conditions

In addition to the [=user-agent style sheet=] in subsection [[[#style-stylesheet]]], we would like to express some style bounds based on the computed style of the element. These should be treated as a [=user-agent style sheet=], too.

For notational convenience, we imagine that the [=computed value=] of an element could be accessed in CSS rules with computed, just like the inherited value of an element can via the [=CSS/inherit=] keyword. Then the following sheet expresses the style bounds: 

          @namespace "http://www.w3.org/1999/xhtml";
          usermedia {
            outline-offset: clamp(0, computed, none); /* No negative outline-offsets. */
            font-weight: clamp(200, computed, none);  /* No font-weights below 200. */
            word-spacing: clamp(0, computed, 0.5em);  /* Word-spacing between 0..0.5em */
            letter-spacing: clamp(-0.05em, computed, 0.2em);  /* Letter spacing between -0.05..0.2em */

            min-height: clamp(1em, computed, none);
            max-height: clamp(none, computed, 3em);
            min-width: clamp(none, computed, calc(fit-content));

            border-width: clamp(none, computed, 1em);

            font-style: if(computed = "normal" or computed = "italic", computed, "normal");
            display: if (computed = "inline-block" or computed = "none", computed, "inline-block");
            cursor: if (computed = "pointer" or computed = "not-allowed", computed, "pointer")
          }

Additionally, some rules apply based on conditions not easily expressible as CSS.

If height is `auto`, then apply:

          @namespace "http://www.w3.org/1999/xhtml";
          usermedia {
            padding-top: clamp(1em, computed, none);
            padding-bottom: calc(padding-top);
          }

If width is `auto`, then apply:

          @namespace "http://www.w3.org/1999/xhtml";
          usermedia {
            padding-left: clamp(none, computed, 5em);
            padding-right: calc(padding-left);
          }

Apply the following sheet, if the element does not have all of the following:

  • A border width of at least `1px`,
  • a color to background-color contrast ratio of at least 3,
  • and alpha of 1.
          @namespace "http://www.w3.org/1999/xhtml";
          usermedia {
            max-width: clamp(none, computed, calc(3 * fit-content));
          }

Algorithms

To add a blocker with a {{ActivationBlockerBlockerReason}} |reason| and an optional flag |expires|:
  1. Let |timestamp| be None.
  2. If |expires|, then let |timestamp| be [=current high resolution time=] plus the [=ActivationBlocker/blocker delay=].
  3. [=list/Append=] an entry to the internal {{ActivationBlocker/[[BlockerList]]}} with |reason| and |timestamp|.
The blocker delay is 500ms.
To add an expiring blocker with a {{ActivationBlockerBlockerReason}} |reason|:
  1. [=Assert=]: |reason| is listed as "expiring" in the [=ActivationBlocker/blocker reason table=].
  2. [=ActivationBlocker/Add a blocker=] with |reason| and true.
To add a temporary blocker with a {{ActivationBlockerBlockerReason}} |reason|:
  1. [=Assert=]: |reason| is listed as "temporary" in the [=ActivationBlocker/blocker reason table=].
  2. [=ActivationBlocker/Add a blocker=] with |reason| and false.
To add a permanent blocker with a {{ActivationBlockerBlockerReason}} |reason|:
  1. [=Assert=]: |reason| is listed as "permanent" in the [=ActivationBlocker/blocker reason table=].
  2. [=ActivationBlocker/Add a blocker=] with |reason| and false.
To remove blockers with {{ActivationBlockerBlockerReason}} |reason| from an |element|:
  1. [=Assert=]: |reason| is listed as "temporary" in the [=ActivationBlocker/blocker reason table=].
  2. [=list/iterate|For each=] |entry| in |element|'s {{ActivationBlocker/[[BlockerList]]}}:
    1. If |entry|'s reason [=string/is|equals=] |reason|, then [=list/remove=] |entry| from |element|'s {{ActivationBlocker/[[BlockerList]]}}.
  3. [=ActivationBlocker/Add a blocker=] with |reason| and true.
To determine an |element|'s blocker:
  1. Let |blockers| be the result of [=list/sorting=] |element|'s {{ActivationBlocker/[[BlockerList]]}} with the [=blocker ordering=] algorithm.
  2. If |blockers| is not [=list/empty=] and |blockers|[0] is [=ActivationBlocker/blocking=], then return |blockers|[0].
  3. Return nothing.
To determine blocker ordering for two blockers |a| and |b|:
  1. Let |really large number| be 99.
  2. [=Assert=]: No order hint in the [=ActivationBlocker/blocker reason table=] is equal to or greater than |really large number|.
  3. If |a| is [=ActivationBlocker/blocking=], then let |a hint| be the order hint of |a|'s [=ActivationBlocker/blocker reason|reason=] in the [=ActivationBlocker/blocker reason table=], otherwise let |a hint| be |really large number|.
  4. If |b| is [=ActivationBlocker/blocking=], then let |b hint| be the order hint of |b|'s [=ActivationBlocker/blocker reason|reason=] in the [=ActivationBlocker/blocker reason table=], otherwise let |b hint| be |really large number|.
  5. Return whether |a hint| is less than or equal to |b hint|.
An element's [=blocker=] list's |entry| is blocking if:
  1. |entry| has no [=ActivationBlocker/blocker timestamp=],
  2. or |entry| has a [=ActivationBlocker/blocker timestamp=], and the [=ActivationBlocker/blocker timestamp=] is greater or equal to the [=current high resolution time=].
This spec maintains blockers as a list {{ActivationBlocker/[[BlockerList]]}}, which may potentially grow indefinitely (since some blocker types simply expire, but are not removed). This structure is chosen for the simplicity of explanation, rather than for efficiency. The details of this blocker structure are not observable except for a handful of algorithms defined here, which should open plenty of opportunities for implementations to handle this more efficiently.
The IntersectionObserver callback implements {{IntersectionObserverCallback}} and runs the following steps:
  1. [=Assert=]: The {{IntersectionObserver}}'s {{IntersectionObserver/root}} is the [=/document=].
  2. Let |entries| be the value of the first callback parameter, the [=/list=] of {{IntersectionObserverEntry}}.
  3. [=Assert=]: |entries| is not [=list/is empty|empty=].
  4. Let |entry| be |entries|'s last [=list/item=].
  5. If |entry|.{{IntersectionObserverEntry/isVisible}}, then:
    1. [=ActivationBlocker/Remove blockers=] with {{ActivationBlockerBlockerReason/intersection_occluded_or_distorted}}.
    2. [=ActivationBlocker/Remove blockers=] with {{ActivationBlockerBlockerReason/intersection_out_of_viewport_or_clipped}}.
  6. Otherwise:
    1. If |entry|.{{IntersectionObserverEntry/intersectionRatio}} >= 1, then:
      1. Let |reason| be {{ActivationBlockerBlockerReason/intersection_occluded_or_distorted}}.
    2. Otherwise:
      1. Let |reason| be {{ActivationBlockerBlockerReason/intersection_out_of_viewport_or_clipped}}.
    3. [=ActivationBlocker/Add a temporary blocker=] with |reason|.
  7. If {{ActivationBlocker/[[IntersectionRect]]}} does not equal |entry|.{{IntersectionObserverEntry/intersectionRect}} then [=ActivationBlocker/add an expiring blocker=] with {{ActivationBlockerBlockerReason/intersection_changed}}.
  8. Set {{ActivationBlocker/[[IntersectionRect]]}} to |entry|.{{IntersectionObserverEntry/intersectionRect}}.
